================================================================================
חלק 1 - האם להשתמש ב-R-TREE?
================================================================================

מה זה R-Tree?
-------------
R-Tree הוא "עץ אינדקס מרחבי" - מבנה נתונים שמארגן אובייקטים לפי המיקום שלהם
במרחב (קו רוחב/אורך), כך שאפשר לשאול במהירות שאלות כמו "מי נמצא בתוך המלבן
הזה?" או "מי קרוב לנקודה הזו?". במקום לעבור על כל הרשומות אחת-אחת (O(n)),
R-Tree מצמצם את החיפוש לאזור הרלוונטי בלבד (בקירוב O(log n)).

האם אני ממליץ עליו? כן - אבל עם הבחנה חשובה:
----------------------------------------------
הרעיון של אינדוקס מרחבי הוא *נכון מאוד* לבעיה שלך. אבל יש שתי דרכים לממש אותו,
ועליך לבחור לפי איפה רצה הלוגיקה:

  אפשרות א' - R-Tree בתוך פייתון (ספריית Rtree):
     מתאים אם ההתאמה רצה בזיכרון בתוך תהליך הפייתון, על קבוצת מועמדים
     שכבר שלפת מה-DB. מימשתי בדיוק את זה ב-matching_service.py.
     יתרון: פשוט, לא תלוי ביכולות ה-DB.
     חיסרון: האינדקס נבנה מחדש בכל קריאה ולא "חי" במסד הנתונים.

  אפשרות ב' (המומלצת לך!) - אינדקס מרחבי של SQL Server (עמודת GEOGRAPHY):
     SQL Server תומך בטיפוס GEOGRAPHY ובאינדקס מרחבי מובנה. האינדקס הזה
     הוא המקבילה ה"תעשייתית" של R-Tree בתוך מסד הנתונים עצמו (מבחינה
     מושגית זו רשת היררכית - B-Tree על תאי-רשת - שמשרתת בדיוק את אותה
     מטרה: חיפוש מרחבי מהיר).
     יתרון: האינדקס נשמר ומתוחזק אוטומטית, השאילתה רצה ב-DB (קרוב לנתונים),
            ומקבלים פונקציות מרחק מובנות (STDistance) "בחינם".
     חיסרון: דורש להגדיר את העמודה והאינדקס פעם אחת (ראה חלק 2).

המלצה סופית:
------------
* לטווח הקצר / לפרויקט לימודי: השאירי את ה-R-Tree של פייתון שמימשתי -
  הוא עובד מצוין ומדגים את העיקרון.
* לטווח הארוך / לייצור: עברי לעמודת GEOGRAPHY + אינדקס מרחבי ב-SQL Server.
  זו הגישה הנכונה כשיהיו הרבה רשומות, כי המסד עושה את העבודה הכבדה.

הערה: בבטון, ה"מרחב" הוא קטן יחסית (ישראל) ומספר ההצעות הפעילות בו-זמנית
לא ענק (בטון פג תוך שעות). לכן גם פתרון פשוט יספיק בהתחלה - אבל האינדקס
המרחבי הופך קריטי ככל שהמערכת גדלה.

חלופות ל-R-Tree (אם לא רוצים אינדקס מרחבי כלל):
------------------------------------------------
1. סינון מקדים לפי אזור (region) + נוסחת Haverside על קבוצה קטנה - זול
   ופשוט. מימשתי גם את זה כשלב ראשון לפני ה-R-Tree.
2. "Geohash" - קידוד הקואורדינטה למחרוזת שבה קידומת משותפת = קרבה. אפשר
   לאנדקס אותה כעמודת טקסט רגילה ולחפש לפי קידומת. פשוט אך פחות מדויק בקצוות.
3. חלוקה לתאי-רשת (grid bucketing) ידנית - מחלקים את הארץ לריבועים ושומרים
   לכל רשומה מספר תא. חיפוש = תא + 8 השכנים. פשוט, אך פחות אלגנטי מ-GEOGRAPHY.


================================================================================
חלק 2 - פקודות SQL Server (הגישה המומלצת: עמודת GEOGRAPHY + אינדקס מרחבי)
================================================================================

אם תבחרי בגישה המומלצת (ב'), אלו הפקודות שצריך להריץ ב-SQL Server.
הן מוסיפות עמודת GEOGRAPHY לשתי הטבלאות (בקשות לקוח והצעות קבלן), ממלאות
אותה מהקואורדינטות הקיימות, ובונות עליה אינדקס מרחבי.

-------- 2.1 הוספת עמודה מרחבית לטבלת בקשות הלקוחות --------

-- מוסיפים עמודה מסוג GEOGRAPHY שתחזיק את הנקודה (lng, lat)
ALTER TABLE ConcreteRequests ADD geo GEOGRAPHY NULL;
GO

-- ממלאים את העמודה מהקואורדינטות הקיימות.
-- שימי לב: ב-SQL Server הסדר הוא Point(lng, lat) - קו אורך קודם!  (SRID 4326 = WGS84, מערכת ה-GPS)
UPDATE ConcreteRequests
SET geo = GEOGRAPHY::Point(lng, lat, 4326)
WHERE lat IS NOT NULL AND lng IS NOT NULL;
GO

-- בונים אינדקס מרחבי על העמודה (זה ה"R-Tree" של SQL Server)
CREATE SPATIAL INDEX SIX_ConcreteRequests_geo
ON ConcreteRequests(geo)
USING GEOGRAPHY_GRID
WITH (GRIDS = (LOW, LOW, MEDIUM, HIGH), CELLS_PER_OBJECT = 16);
GO

-------- 2.2 הוספת עמודה מרחבית לטבלת הצעות הקבלנים --------

ALTER TABLE ContractorConcreteRequests ADD geo GEOGRAPHY NULL;
GO

UPDATE ContractorConcreteRequests
SET geo = GEOGRAPHY::Point(lng, lat, 4326)
WHERE lat IS NOT NULL AND lng IS NOT NULL;
GO

CREATE SPATIAL INDEX SIX_ContractorOffers_geo
ON ContractorConcreteRequests(geo)
USING GEOGRAPHY_GRID
WITH (GRIDS = (LOW, LOW, MEDIUM, HIGH), CELLS_PER_OBJECT = 16);
GO

-------- 2.3 שאילתה לדוגמה: מצא הצעות קבלנים ברדיוס 30 ק"מ מבקשת לקוח --------

-- נניח שיש לנו את הנקודה של הלקוח (@customerGeo).
-- STDistance מחזיר מרחק במטרים. 30 ק"מ = 30000 מטר.
DECLARE @customerGeo GEOGRAPHY = (
    SELECT geo FROM ConcreteRequests WHERE request_id = @requestId
);

SELECT TOP 50
       o.request_id,
       o.contractor_id,
       o.quantity,
       o.geo.STDistance(@customerGeo) AS distance_meters   -- המרחק בפועל
FROM   ContractorConcreteRequests AS o
WHERE  o.geo IS NOT NULL
  AND  o.geo.STDistance(@customerGeo) <= 30000             -- בתוך 30 ק"מ
  AND  (o.expiry_time IS NULL OR o.expiry_time > GETDATE())-- עוד לא פג
ORDER BY distance_meters ASC;                              -- הקרוב ראשון
GO

-- הערה: בזכות האינדקס המרחבי, STDistance משתמש באינדקס ולא סורק את כל הטבלה.

-------- 2.4 (אופציונלי) שמירה אוטומטית של geo בכל הוספה/עדכון --------

-- אפשר Trigger שממלא את geo אוטומטית מ-lat/lng, כך שלא צריך לזכור ידנית:
CREATE TRIGGER trg_ConcreteRequests_geo
ON ConcreteRequests
AFTER INSERT, UPDATE
AS
BEGIN
    SET NOCOUNT ON;
    UPDATE cr
    SET cr.geo = GEOGRAPHY::Point(i.lat, i.lng, 4326)
    FROM ConcreteRequests cr
    JOIN inserted i ON cr.request_id = i.request_id
    WHERE i.lat IS NOT NULL AND i.lng IS NOT NULL;
END;
GO

(אם תרצי, אפשר ליצור Trigger מקביל גם ל-ContractorConcreteRequests.)


================================================================================
חלק 3 - אסטרטגיית שליחת ההתראות: שידור לכולם מול סדרתי עם המתנה
================================================================================

השאלה שלך: יש 5 לקוחות מתאימים -
  (א) לשלוח לכולם, והראשון שתופס זוכה? או
  (ב) לשלוח לעדיפות הגבוהה, לחכות ~3 דקות, ואם לא תפס - לעבור לבא בתור?

ניתוח שתי האפשרויות:
--------------------
אפשרות א' - "שידור" (Broadcast):
   + מהיר מאוד - מישהו כנראה יענה כמעט מיד.
   + פשוט לממש.
   - לא הוגן: לקוח שמחכה שבועיים עלול להפסיד ללקוח מהיר-אצבע שמחכה יום.
   - "מרוץ": כמה לקוחות מאשרים כמעט יחד -> צריך לטפל במצב שכבר נתפס, וחלקם
     מתאכזבים ("רצתי לאשר וזה כבר נלקח").
   - שורף אמון: לקוחות מקבלים התראות על הצעות ש"נחטפות" מהר.

אפשרות ב' - "סדרתי עם המתנה" (Sequential with timeout):
   + הוגן: מכבד את סדר העדיפות (מי שמחכה הכי הרבה מקבל הזדמנות ראשונה).
   + חוויה נקייה: רק לקוח אחד מוזמן בכל רגע, אין "מרוצים".
   - איטי יותר: אם הראשון לא עונה, מבזבזים את חלון ההמתנה (3 דק') עד הבא.
   - בבטון, 3 דק' כפול כמה לקוחות עלול להצטבר לזמן יקר (הבטון מתקשה!).

ההמלצה שלי: גישה היברידית (וזו ברירת המחדל בקוד שמימשתי)
--------------------------------------------------------
המלצתי היא להעדיף את האפשרות הסדרתית (ב') כברירת מחדל, אבל לכוונן אותה
לאופי הדחוף של הבטון, כך:

  1. חלון המתנה קצר ודינמי: לא 3 דקות קבועות. ככל שזמן התפוגה של הבטון
     קרוב יותר, מקצרים את חלון ההמתנה (למשל 90 שניות במקום 180).
  2. שילוב "סדרתי בקבוצות קטנות" (Sequential batching): במקום לקוח אחד
     בכל פעם, לשלוח ל-2-3 הלקוחות בעדיפות הגבוהה ביותר יחד, ואז להמתין.
     זה מאזן בין הוגנות (עדיין לפי עדיפות) למהירות (סיכוי טוב שאחד מהם יענה).
  3. נפילה ל"שידור" כשהזמן אוזל: אם נשארו פחות מ-X דקות עד שהבטון מתקשה,
     עוברים אוטומטית לשידור לכל המתאימים - כי אז מהירות > הוגנות.

למה זה הכי טוב לבטון? כי בבטון יש *שני* ערכים מתנגשים: הוגנות (מי מחכה הכי
הרבה) ודחיפות (הבטון מתקשה). גישה קבועה אחת לא יכולה לאזן בין השניים -
הגישה ההיברידית מתאימה את עצמה למצב: רגועה כשיש זמן, אגרסיבית כשהזמן אוזל.

איך זה ממומש בקוד:
-----------------
בקובץ notification_service.py מימשתי את שתי האסטרטגיות:
  - dispatch_offer_to_candidates(...) - הנקודה הראשית, בוחרת אסטרטגיה.
  - _dispatch_sequential(...) - סדרתי עם המתנה ובדיקת "נתפס".
  - _dispatch_broadcast(...) - שידור לכולם.
ההחלפה ביניהן נעשית דרך ההגדרה DISPATCH_STRATEGY ב-.env (sequential/broadcast),
וזמן ההמתנה דרך SEQUENTIAL_WAIT_SECONDS. כך תוכלי לנסות את שתיהן בקלות,
ולהוסיף בהמשך את הלוגיקה הדינמית (קיצור החלון לפי זמן התפוגה).


================================================================================
חלק 4 - נוסחת העדיפות (זמן המתנה מול זמן נסיעה)
================================================================================

הדרישה שלך: לקוח שמחכה שבועיים עם נסיעה של חצי שעה צריך לקבל עדיפות גבוהה
יותר מלקוח שמחכה שבוע עם נסיעה של 10 דקות - כי ההמתנה הארוכה גוברת.

הנוסחה שמימשתי (בפונקציה calculate_priority):

    ציון = (משקל_המתנה * ימי_המתנה) − (משקל_נסיעה * דקות_נסיעה)

עם משקלי ברירת המחדל (1.0 להמתנה, 0.3 לנסיעה), הדוגמה שלך נותנת:
    לקוח א': מחכה 14 יום, נסיעה 30 דק'  ->  14*1.0 − 30*0.3 = 14 − 9 = 5.0
    לקוח ב': מחכה 7 יום,  נסיעה 10 דק'  ->  7*1.0  − 10*0.3 = 7  − 3 = 4.0
    => לקוח א' מקבל עדיפות גבוהה יותר. בדיוק כפי שרצית.

ההמתנה (ביחידות של ימים) מקבלת משקל גדול פי כמה מהנסיעה (ביחידות דקות),
ולכן היא ה"גורם השולט" - אבל הנסיעה עדיין משפיעה ומבדילה בין לקוחות עם
המתנה דומה. את המשקלים אפשר לכוונן ב-.env (PRIORITY_WAIT_WEIGHT,
PRIORITY_TRAVEL_WEIGHT) בלי לגעת בקוד.

הערה: אפשר בעתיד להוסיף לנוסחה גם את "דחיפות התפוגה" של ההצעה (כמה דקות
נשארו לבטון) וגם את התאמת הכמות - מבנה הקוד מאפשר זאת בקלות.


================================================================================
חלק 5 - הסבר מפורט על כל קובץ ופונקציה בשכבת ה-service
================================================================================

הקבצים נמצאים בתיקייה python-service/service/ (להעתקה למאגר projectBeton).

--- geo_config.py ---
ריכוז כל המפתחות והקבועים במקום אחד (נטענים מ-.env):
  • GOOGLE_MAPS_API_KEY - המפתח ל-Geocoding ול-Routes.
  • MATCH_RADIUS_KM - רדיוס חיפוש מקסימלי.
  • QUANTITY_TOLERANCE - כמה עודף כמות עדיין נחשב התאמה.
  • PRIORITY_WAIT_WEIGHT / PRIORITY_TRAVEL_WEIGHT - משקלי נוסחת העדיפות.
  • DISPATCH_STRATEGY / SEQUENTIAL_WAIT_SECONDS - אסטרטגיית ההפצה.

--- geocoding_service.py ---  (זה החלק שביקשת: כתובת -> קואורדינטות -> DB)
  • geocode_address(address) - שולח כתובת ל-Google Geocoding ומחזיר (lat,lng).
  • geocode_and_save_customer_request(request_id) - טוען בקשת לקוח, ממיר את
    הכתובת שלה, ושומר lat/lng ב-DB.
  • geocode_and_save_contractor_offer(offer_id) - אותו דבר עבור הצעת קבלן.
  כך גם לקוח וגם קבלן מקבלים קואורדינטות מהכתובת - בדיוק כפי שביקשת.

--- region_service.py ---  (סינון גס לפי אזור)
  • get_region_from_coordinates(lat,lng) - Reverse Geocoding: קואורדינטות
    -> שם מחוז/עיר (דרך Google).
  • get_nearby_regions(region) - מחזיר אזורים סמוכים, לסינון ראשוני זול
    לפני החישוב היקר.

--- matching_service.py ---  (הלב!)
  • calculate_distance(...) - מרחק אווירי בק"מ בנוסחת Haversine.
  • travel_time_minutes(...) - זמן נסיעה אמיתי בכביש דרך Google Routes API
    (מתחשב בתנועה). מחזיר None אם אין מפתח, ואז מסתמכים על מרחק אווירי.
  • calculate_priority(waiting_days, travel_minutes) - נוסחת העדיפות (חלק 4).
  • _concrete_types_overlap(...) - בודק חפיפת סוג בטון (לפי מטרה - ראה הערה
    על המודל למטה).
  • _quantity_fits(...) - בודק שהקבלן מציע מספיק (ולא יותר מדי).
  • _build_rtree(...) / _query_rtree(...) - בניית אינדקס R-Tree על המועמדים
    ושאילתה מרחבית סביב נקודה. אם Rtree לא מותקן -> נפילה רכה לסריקה לינארית.
  • find_matches_for_contractor(offer_id) - מהצעת קבלן -> רשימת לקוחות
    מתאימים ממוינת לפי עדיפות (זה התרחיש מהשאלה שלך עם 5 הלקוחות).
  • find_matches_for_customer(request_id) - מבקשת לקוח -> הצעות קבלנים
    מתאימות ממוינות לפי קרבה, תוך סינון הצעות שפגו.

  הצנרת (pipeline): סינון אזור (זול) -> R-Tree (מהיר) -> סינון מדויק
  (מרחק/סוג/כמות/תפוגה) -> Google Routes (יקר, רק לשורדים) -> עדיפות + מיון.
  הסדר הזה מצמצם הרבה בשלבים הזולים, כדי לקרוא ל-Google מעט ככל האפשר.

  הערה חשובה על המודל: בקשת הלקוח שומרת רק purpose_id (מטרה), בעוד הצעת
  הקבלן מצביעה על Concrete_type מלא (חוזק/סומך/אבן/מטרה). המכנה המשותף הוא
  ה-Purpose בלבד, ולכן חפיפת "סוג הבטון" נבדקת לפי מטרה. שיפור מומלץ:
  להוסיף לבקשת הלקוח גם strength_id/stone_size_id כדי לאפשר התאמה מדויקת.

--- notification_service.py ---  (התראות + אסטרטגיית הפצה)
  • send_sms(phone, message) - שליחה פלאגבילית: Twilio אם מוגדר, אחרת

  • send_match_notification_to_customer / _to_contractor - הודעות מוכנות.
  • send_expiry_warning - התראה לקבלן שההצעה עומדת לפוג.
  • dispatch_offer_to_candidates + _dispatch_sequential + _dispatch_broadcast
    - שתי אסטרטגיות ההפצה (חלק 3).

הסבר על כל דף (מה הוא עושה ולמי הוא מיועד):
  • דף הבית: שער הכניסה. מסביר את הרעיון ומפנה לשני המסלולים. אם המשתמש
    כבר מחובר - הכרטיסים מובילים ישר לפעולה לפי תפקידו.
  • הרשמה: יצירת לקוח/קבלן חדש בשרת (שם + טלפון). אחרי הרשמה המשתמש מחובר
    אוטומטית ומועבר לדף הפעולה שלו.
  • התחברות: איתור משתמש קיים לפי טלפון + תפקיד. (בשרת אין סיסמאות עדיין -
    ראה הערה למטה.)
  • הגשת בקשה מלקוח: בחירת מטרה, כמות, אזור, מיקום (כולל כפתור "אתר מיקומי") -
    ושליחה ל-POST /concrete-requests/.
  • פרסום הצעה מקבלן: בחירת סוג בטון (עם תווית קריאה הנבנית מטבלאות העזר),
    כמות, זמן תפוגה, אזור, מיקום - ושליחה ל-POST /contractor-offers/.

הערה על הרשמה/התחברות: בשרת הנוכחי הטבלאות Customers/Contractors מכילות רק
שם וטלפון, ללא סיסמה. לכן "הרשמה" = יצירת רשומה, ו"התחברות" = איתור לפי
טלפון. זה תואם בדיוק את השרת הקיים. כשתוסיפי אבטחת סיסמאות (למשל JWT),
צריך לעדכן רק את AuthContext.jsx - שאר האתר לא ישתנה.

חלק 7 - איך הכל מתחבר יחד + צעדים הבאים
================================================================================

זרימה מלאה (תרחיש לדוגמה):
  1. קבלן נרשם (React -> POST /contractors/) ומפרסם הצעת עודפי בטון עם
     כתובת, כמות וזמן תפוגה (POST /contractor-offers/).
  2. בשרת, geocoding_service ממיר את הכתובת לקואורדינטות ושומר ב-DB.
  3. matching_service.find_matches_for_contractor מוצא לקוחות מתאימים:
     סינון אזור -> R-Tree -> סינון מרחק/סוג/כמות -> Google Routes -> עדיפות.
  4. notification_service מפיץ את ההצעה ללקוחות לפי האסטרטגיה (סדרתי/שידור).
  5. לקוח מקבל SMS, נכנס לאתר ומאשר. ההצעה "נתפסת".

צעדים הבאים מומלצים (לפי סדר חשיבות):
  1. הוספת endpoints בשרת שמפעילים את השירותים (כרגע מימשנו את ה-services;
     צריך controller קטן שקורא ל-find_matches ו-dispatch).
  2. הוספת שדה "סטטוס" להצעה/בקשה (פתוח / נתפס / פג) - דרוש ל-is_taken.
  3. מעבר לעמודת GEOGRAPHY + אינדקס מרחבי ב-SQL Server (חלק 2).
  4. הוספת אבטחת התחברות אמיתית (סיסמאות + JWT) - עדכון רק ב-AuthContext.
  5. שיפור מודל ההתאמה: להוסיף לבקשת הלקוח גם strength/stone_size לדיוק.
  6. חיבור ספק SMS אמיתי (Twilio) דרך משתני הסביבה.

בהצלחה! כל הקוד מתועד בעברית שורה-שורה, כך שתוכלי לעקוב ולהרחיב בקלות.
